This demo walks through a full DevOps cycle: modifying application code, triggering a CI/CD pipeline, and watching the change propagate to the test and production environments.

== Scenario

The `line-dashboard` application displays IoT sensor data. You will make a visible UI change, rebuild the image via Tekton, and observe the update roll out through the GitOps pipeline.

== Step 1: View the current dashboard

Open the Line Dashboard and note the current title:

[source,bash,subs=attributes+]
----
echo "https://$(oc get route line-dashboard -n industrial-edge-tst-all -o jsonpath='{.spec.host}')"
----

The page header reads *IoT Dashboard*.

== Step 2: Edit the application code

Open the Gitea web interface and navigate to the `manuela-dev` repository (if cloned into Gitea). Find the file:

`components/iot-frontend/src/app/app.component.html`

Change the dashboard title:

[source,diff]
----
-  IoT Dashboard
+  IoT Dashboard - DEVOPS was here!
----

Commit the change directly in the Gitea web interface.

== Step 3: Start the CI/CD pipeline

Trigger the `build-and-test-iot-frontend` pipeline that will:

. Clone the source code from Gitea
. Build a new container image with the change
. Push the image to the OpenShift internal registry
. Deploy the new image to the `industrial-edge-tst-all` test namespace
. Create a Pull Request in Gitea for promotion to production

[source,bash]
----
oc create -f examples/helm/components/industrial-edge-pipelines/extra/build-and-test-iot-frontend.yaml \
  -n industrial-edge-ci
----

== Step 4: Monitor the pipeline

Watch the pipeline execution in real-time:

[source,bash]
----
oc get pipelineruns -n industrial-edge-ci -w
----

Or check the OpenShift Console under *Pipelines* → *PipelineRuns* in the `industrial-edge-ci` namespace.

The pipeline stages are:

[cols="1,2", options="header"]
|===
| Stage | Action

| `git-clone`
| Clones the application source from Gitea

| `build`
| Builds the container image using Buildah

| `push`
| Pushes the image to the internal OpenShift registry

| `deploy-test`
| Updates the image tag in the test environment

| `create-pr`
| Creates a Pull Request in Gitea for production promotion
|===

== Step 5: Verify in test environment

After the pipeline completes, ArgoCD will refresh the `industrial-edge-tst` application and deploy the new image.

Check the test Line Dashboard — you should see the updated title: *IoT Dashboard - DEVOPS was here!*

You may need to clear your browser cache or do a hard refresh (Ctrl+Shift+R) to see the change.

== Step 6: Promote to production

Open Gitea and review the Pull Request created by the pipeline. The PR updates the image tag in the production (stormshift) configuration.

. Verify the change is correct in the test environment
. Merge the PR in Gitea
. ArgoCD will automatically deploy the change to the production factory namespaces

[source,bash]
----
oc get application field-content-industrial-edge-stormshift -n openshift-gitops -w
----

== Key takeaways

* *Automated build and test* — Tekton pipelines ensure consistent, repeatable builds
* *Image promotion* — Images flow from internal registry → test → production via GitOps
* *Pull Request workflow* — Production changes require explicit approval (merge)
* *Separation of concerns* — Developers commit code; pipelines handle builds; ArgoCD handles deployment
* *Traceability* — Every production deployment traces back to a specific code commit and pipeline run
