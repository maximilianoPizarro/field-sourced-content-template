This section covers the *Industrial Edge* validated pattern — an end-to-end IoT manufacturing solution that demonstrates real-time sensor monitoring, event streaming, machine learning inference, and GitOps-driven operations on Red Hat OpenShift.

The pattern is derived from the https://validatedpatterns.io/patterns/industrial-edge/[Red Hat Validated Patterns^] project and has been adapted for single-cluster deployment within this workshop environment.

== Solution overview

The Industrial Edge pattern addresses a common manufacturing challenge: bringing AI/ML capabilities to the factory floor while maintaining centralized control over data, models, and application lifecycle.

image::https://validatedpatterns.io/images/industrial-edge/overview.png[Industrial Edge Overview,800]

The solution provides:

* *Real-time sensor monitoring* — IoT sensors publish temperature and vibration data via MQTT
* *Event streaming* — AMQ Broker and Kafka transport sensor data across the platform
* *ML-based anomaly detection* — Trained models perform live inference on sensor readings
* *Data lake* — Sensor data is stored in S3-compatible storage (MinIO) for model training
* *CI/CD pipelines* — Tekton pipelines automate image builds and GitOps-based deployments
* *GitOps operations* — Configuration and application changes flow through Git and ArgoCD

== Architecture (single-cluster adaptation)

This workshop deploys all components on a single OpenShift cluster using separate namespaces to simulate the factory/datacenter split:

[cols="1,2,1", options="header"]
|===
| Namespace | Role | Components

| `industrial-edge-tst-all`
| Development/test environment (full IoT stack)
| Machine sensors, AMQ Broker, Kafka, Camel K, Line Dashboard, Anomaly Detection

| `industrial-edge-stormshift-messaging`
| Factory messaging layer
| Kafka cluster, AMQ Broker, MQTT-to-Kafka bridge

| `industrial-edge-stormshift-machine-sensor`
| Factory sensor simulation
| Machine sensor pods (temperature + vibration)

| `industrial-edge-stormshift-line-dashboard`
| Factory visualization
| IoT Line Dashboard web app

| `industrial-edge-data-lake`
| Central data lake
| Kafka cluster, Kafka-to-S3 integration (Camel K), KafkaMirrorMaker2

| `industrial-edge-ml-workspace`
| ML workspace and pipelines
| OpenShift AI workbench, model training pipelines, S3 data connections

| `industrial-edge-ci`
| CI/CD pipelines
| Tekton pipelines for building and deploying IoT component images

| `industrial-edge-inference`
| Model inference
| Anomaly detection inference service (KServe)
|===

== Technology stack

[cols="1,3", options="header"]
|===
| Technology | Purpose

| *Red Hat AMQ Broker*
| MQTT broker for IoT sensor data ingestion

| *AMQ Streams (Kafka)*
| Durable event streaming backbone for sensor data and data mirroring

| *Apache Camel K*
| Lightweight integration: MQTT-to-Kafka bridge, Kafka-to-S3 data lake ingestion

| *Red Hat OpenShift AI*
| ML model training (Jupyter notebooks), model serving (KServe), data science pipelines

| *MinIO*
| S3-compatible object storage for the data lake (replaces OpenShift Data Foundation)

| *OpenShift Pipelines (Tekton)*
| CI/CD for building sensor, dashboard, and consumer images; model deployment

| *OpenShift GitOps (ArgoCD)*
| Declarative application delivery and configuration management
|===

== Data flows

=== Northbound (sensor → datacenter)

Sensor data flows from the factory floor to the central data lake:

. *Machine sensors* generate temperature and vibration readings
. Readings are published via *MQTT* to the *AMQ Broker*
. *Camel K* bridges MQTT messages into *Kafka* topics
. *KafkaMirrorMaker2* replicates factory Kafka topics to the central *data lake* Kafka cluster
. *Camel K* writes from Kafka to *MinIO S3* buckets for long-term storage and ML training

=== Southbound (datacenter → factory)

Models and configuration flow from the datacenter back to the factory:

. Data scientists train anomaly detection models in *OpenShift AI* notebooks
. Trained model artifacts are committed to *Git*
. *Tekton pipelines* build model images and push them to the internal registry
. *ArgoCD* deploys updated models and configurations to factory namespaces via GitOps

=== Live inference

At the factory edge, the anomaly detection model runs in real-time:

. The *inference service* (KServe) receives sensor readings
. It applies the trained ML model to detect anomalies
. Results are displayed on the *Line Dashboard*
. Anomaly alerts appear every 2–3 minutes under normal operation

== Cluster sizing

This single-cluster deployment adds approximately *8 vCPU* and *16 Gi memory* to the base workshop footprint. The key resource consumers are:

[cols="1,1,1", options="header"]
|===
| Component | CPU | Memory

| Kafka clusters (2)
| 2 vCPU
| 4 Gi

| OpenShift AI (RHODS)
| 2 vCPU
| 4 Gi

| AMQ Broker
| 0.5 vCPU
| 1 Gi

| Camel K integrations
| 1 vCPU
| 2 Gi

| MinIO
| 0.5 vCPU
| 1 Gi

| Sensors + Dashboard
| 0.5 vCPU
| 1 Gi

| Tekton pipelines (idle)
| 0.5 vCPU
| 1 Gi

| ML inference
| 1 vCPU
| 2 Gi
|===
