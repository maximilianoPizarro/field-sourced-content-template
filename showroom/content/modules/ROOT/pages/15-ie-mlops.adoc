= Step 13: Enable Temperature Sensors via GitOps

Temperature sensors are deployed but *disabled* by default. In this step you will enable them by editing a `ConfigMap` — and watch the change propagate through the system.

== Before: verify temperature is off

Check the current sensor config:

[source,bash]
----
oc get configmap machine-sensor-1 -n industrial-edge-tst-all \
  -o jsonpath='{.data.SENSOR_TEMPERATURE_ENABLED}'
echo
----

Output: `false`

Open the link:https://line-dashboard-industrial-edge-tst-all.{cluster_domain}[Dev Line Dashboard^] → *Realtime Data*. Notice: only vibration charts have data. Temperature is flat/empty.

== Action: enable temperature

Patch the ConfigMaps directly to enable temperature on both dev sensors:

[source,bash]
----
oc patch configmap machine-sensor-1 -n industrial-edge-tst-all \
  --type merge -p '{"data":{"SENSOR_TEMPERATURE_ENABLED":"true"}}'

oc patch configmap machine-sensor-2 -n industrial-edge-tst-all \
  --type merge -p '{"data":{"SENSOR_TEMPERATURE_ENABLED":"true"}}'
----

Restart the sensor pods to pick up the new config:

[source,bash]
----
oc rollout restart deploy/machine-sensor-1 -n industrial-edge-tst-all
oc rollout restart deploy/machine-sensor-2 -n industrial-edge-tst-all
----

== Verify the change

Wait for the pods to restart:

[source,bash]
----
oc rollout status deploy/machine-sensor-1 -n industrial-edge-tst-all --timeout=30s
----

Check sensor logs — you should now see *temperature* readings alongside vibration:

[source,bash]
----
oc logs deploy/machine-sensor-1 -n industrial-edge-tst-all --tail=5
----

== Observe the result

Go back to the link:https://line-dashboard-industrial-edge-tst-all.{cluster_domain}[Dev Line Dashboard^] → *Realtime Data*.

After 15–20 seconds, *temperature* charts will start populating alongside vibration.

Check that the new data type flows through Kafka:

[source,bash]
----
oc exec -n industrial-edge-tst-all dev-cluster-dev-0 -- \
  bin/kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic iot-sensor-sw-temperature \
  --max-messages 3
----

== How this works in production (GitOps)

What you just did with `oc patch` is the *imperative* approach. In production, the same change flows through *GitOps*:

. A developer edits the ConfigMap template in Git
. ArgoCD detects the drift and syncs the change
. Pods pick up the new ConfigMap automatically

The config templates live in:
`examples/helm/components/industrial-edge-tst/templates/machine-sensor/machine-sensor-1-configmap.yaml`

== Undo (optional)

To disable temperature again:

[source,bash]
----
oc patch configmap machine-sensor-1 -n industrial-edge-tst-all \
  --type merge -p '{"data":{"SENSOR_TEMPERATURE_ENABLED":"false"}}'
oc patch configmap machine-sensor-2 -n industrial-edge-tst-all \
  --type merge -p '{"data":{"SENSOR_TEMPERATURE_ENABLED":"false"}}'
oc rollout restart deploy/machine-sensor-1 deploy/machine-sensor-2 -n industrial-edge-tst-all
----

== What you observed

* Configuration change via ConfigMap → pod restart → immediate effect
* New sensor data type appearing on the dashboard in seconds
* Same data flowing through Kafka topics end-to-end
