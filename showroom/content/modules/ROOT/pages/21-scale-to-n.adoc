The validated Industrial Edge pattern runs as a *fixed* set of namespaces.
The *Scale-to-N* capability turns it into a **self-service platform** where any user can spin up an isolated, production-grade IoT stack on demand -- directly from Red Hat Developer Hub.

== The problem

A single Industrial Edge deployment proves the architecture. But what happens when:

* Multiple teams need their own isolated environments?
* A partner demo requires a fresh instance per attendee?
* You need to test a Camel K integration change without affecting production sensors?

Manually creating namespaces, Kafka clusters, and ArgoCD Applications does not scale.

== The solution: Software Template as platform API

The `industrial-edge` Software Template exposes the validated architecture as a **form** in Developer Hub. Each execution produces a fully autonomous edge instance with its own:

[cols="1,2", options="header"]
|===
| Resource | Details

| Namespace
| `<owner>-<instance-name>` (e.g. `user1-edge-factory-1`)

| Machine Sensors
| 1, 2, 4 or 8 Spring Boot simulators (configurable)

| AMQ Broker
| MQTT broker receiving sensor telemetry

| Kafka Cluster (KRaft)
| 1- or 3-node cluster in the instance namespace

| Camel K Integration
| MQTT -> Kafka bridge with `camel:kafka` + `camel:paho`

| Line Dashboard
| Real-time sensor visualization with its own Route

| ArgoCD Application
| Auto-sync, auto-prune, namespace auto-creation

| Catalog Entities
| System + 5 Components + 2 APIs registered in the catalog

| Kafka Streams Console
| Link to the shared StreamsHub console
|===

== Architecture: from 1 to N

The 3D scene above illustrates the pattern. The *platform layer* (Developer Hub, Gitea, ArgoCD) is shared. Each user instance is fully isolated:

* **Namespace isolation** -- every instance lives in `<owner>-<name>`, with dedicated RBAC
* **Kafka isolation** -- each instance runs its own KRaft cluster (no cross-contamination)
* **GitOps isolation** -- each instance has its own Git repository and ArgoCD Application
* **Catalog isolation** -- each instance registers its own System, Components, and APIs

== Step-by-step: deploy a new edge instance

=== 1. Open Developer Hub

Navigate to link:https://backstage-developer-hub-developer-hub.{cluster_domain}[Developer Hub^] and click **Create...** in the left sidebar.

=== 2. Select the Industrial Edge template

Find *"Industrial Edge: IoT Manufacturing Platform"* and click **Choose**.

=== 3. Fill the form

[cols="1,2,1", options="header"]
|===
| Field | Description | Example

| Instance Name
| Unique identifier for this edge
| `edge-factory-1`

| Description
| Free-text label
| `Production line Buenos Aires`

| User
| Your OpenShift username (auto-detected)
| `user1`

| Number of Machine Sensors
| Sensor simulators to deploy
| `2`

| Kafka Broker Replicas
| 1 for dev, 3 for HA
| `3`

| Enable Anomaly Detection
| Deploys ML InferenceService
| `false`

| Cluster Domain
| OpenShift apps domain
| `apps.cluster-xxx.dynamic.redhatworkshops.io`
|===

=== 4. Review and Create

Click **Next Step**, review the summary, then **Create**. The scaffolder executes 5 steps:

[source,text]
----
1. Generate Skeleton       → Renders all manifests from the skeleton template
2. Publish to Gitea        → Creates repo ws-user1/edge-factory-1
3. Create ArgoCD App       → POST to K8s API → Application CR in openshift-gitops
4. Register in Catalog     → Backstage discovers System + Components + APIs
5. Send Email Notification → Mailpit receives the deployment confirmation
----

=== 5. Watch it deploy

Within 2-3 minutes ArgoCD syncs the Application:

[source,bash]
----
# Watch namespace creation and pod startup
oc get pods -n user1-edge-factory-1 -w

# Check ArgoCD status
oc get application user1-edge-factory-1 -n openshift-gitops -o jsonpath='{.status.sync.status}'
----

== Clean up an instance

Use the *Remove Component* template in Developer Hub:

. Click **Create...** → select **Remove Component (Cleanup)**
. Select template type: *Industrial Edge*
. Enter the instance name and owner
. Execute — the template:
  * Deletes the Gitea repository
  * Cascade-deletes the ArgoCD Application (removes namespace and all resources)
  * Unregisters all catalog entities (System + Components + APIs)
  * Sends a decommission notification email

== Scaling considerations

[cols="1,3", options="header"]
|===
| Factor | Guidance

| Cluster resources
| Each instance requires ~2 vCPU / 4 GiB RAM (3-node Kafka). Use `kafkaReplicas: 1` for dev to halve requirements.

| Max instances
| Limited by cluster capacity. A 48-vCPU / 192 GiB worker pool supports ~15-20 concurrent 3-node instances.

| Namespace quotas
| Consider adding `ResourceQuota` and `LimitRange` to the skeleton for multi-tenant clusters.

| Kafka isolation
| Each instance runs its own Kafka cluster. For shared Kafka, modify the skeleton to point to the central cluster.

| Sensor count
| 1-8 sensors per instance. Each sensor uses ~100m CPU / 128Mi RAM.

| Storage
| Each Kafka broker requests a PVC. Ensure the cluster has sufficient `StorageClass` capacity.
|===

== Key technical decisions

* **K8s API proxy for ArgoCD** -- Applications are created via the Kubernetes API (`/proxy/k8s-api/apis/argoproj.io/v1alpha1/...`) using the Developer Hub ServiceAccount. No ArgoCD-specific tokens needed.
* **ApplicationSet exclusion** -- Repos with `argocd/application.yaml` are excluded from the global ApplicationSet to prevent duplicate Application creation.
* **S2I for Camel K** -- The `IntegrationPlatform` uses `publishStrategy: S2I` to leverage native OpenShift builds instead of Jib.
* **Probe tuning** -- Spring Boot sensors have `initialDelaySeconds: 60s` (liveness) to accommodate slow JVM startup.
