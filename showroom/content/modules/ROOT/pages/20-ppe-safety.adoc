== Overview

The *PPE Safety Detection* module extends the NeuroFace platform with real-time Personal Protective Equipment monitoring. Webcam frames captured by the Angular frontend are sent to the FastAPI backend, which forwards them to a *YOLOv8n* serving endpoint for object detection. The backend classifies compliance against expected PPE items (hardhat, safety vest, goggles), optionally calls the *Granite 3.1 LLM* for natural-language safety analysis, and publishes each detection event to the Kafka topic `cv.ppe.detections`. A Jupyter notebook consumes these events for visualization and compliance dashboards.

== Architecture

[cols="1,2,1",options="header"]
|===
| Component | Role | Namespace

| *NeuroFace Frontend*
| Angular UI — captures webcam frames via WebRTC and sends base64 images to the PPE API
| `neuroface`

| *NeuroFace Backend*
| FastAPI — `/ppe/detect` endpoint decodes image, calls YOLO, classifies compliance, calls LLM, publishes to Kafka
| `neuroface`

| *YOLO PPE Serving*
| Python HTTP server running Ultralytics YOLOv8n — receives raw JPEG bytes, returns bounding boxes and class labels
| `neuroface`

| *Granite 3.1 2B LLM*
| KServe InferenceService — generates natural-language safety analysis when a PPE violation is detected
| `neuroface` / `sandbox-shared-models`

| *Kafka Topic*
| `cv.ppe.detections` on Strimzi `cdc-cluster` with SASL_SSL + SCRAM-SHA-512
| `kafka-cdc`

| *Kafka DLQ*
| `dlq.ppe-errors` — dead letter queue for failed processing
| `kafka-cdc`

| *Jupyter Workbench*
| OpenShift AI Notebook (`ppe-yolo-detection.ipynb`) — consumes Kafka events, plots compliance charts
| `neuroface`
|===

== Data Flow

Each webcam frame follows this path:

. *Capture* — The Angular frontend captures a frame from the browser webcam via WebRTC
. *Send to Backend* — The frame is base64-encoded and `POST`-ed to `/ppe/detect`
. *YOLO Inference* — The backend re-encodes to JPEG and forwards to the YOLO serving endpoint (`/v1/predict`), receiving bounding boxes with class names and confidence scores
. *Compliance Check* — The backend compares detected objects against expected PPE classes (`hardhat`, `safety-vest`, `goggles`) and assigns a status: `compliant`, `violation`, or `no_persons`
. *LLM Analysis* (on violation) — When PPE is missing, the backend calls Granite 3.1 with a structured prompt to generate a safety analysis
. *Kafka Publish* — The full detection event (objects, status, LLM analysis, timestamp) is published to `cv.ppe.detections` using `confluent-kafka` with SASL_SSL
. *Notebook Consumption* — The Jupyter notebook consumes events from Kafka and renders compliance dashboards with matplotlib

== Granite LLM Integration

When the compliance check returns `violation`, the backend constructs a prompt:

[source,text]
----
You are a workplace safety AI assistant. Analyze these YOLO detections from a factory floor camera:
- Persons detected: 1
- Objects detected: person, safety-vest
- Expected PPE: hardhat, safety-vest, goggles

Determine if each operator is in compliance with safety protocol.
List any missing PPE and explain the risk in 2-3 sentences.
----

The LLM response is included in the Kafka event under `llm_analysis` and displayed in the frontend UI.

== Jupyter Notebook

The `ppe-yolo-detection.ipynb` notebook (seeded into the workbench on first boot) provides:

* *YOLO Health Check* — verifies the serving endpoint is reachable
* *PPE Detection & Visualization* — sends test images and draws bounding boxes with matplotlib
* *Kafka Consumer* — reads events from `cv.ppe.detections` and displays them as a DataFrame
* *Compliance Dashboard* — pie charts for PPE status distribution, bar charts for missing items frequency
* *Granite LLM Analysis* — sends detection summaries to the LLM and displays the safety report

Access the notebook at link:https://rhods-dashboard-redhat-ods-applications.{cluster_domain}/projects/neuroface[OpenShift AI -> neuroface project^].

== Security

* *Kafka authentication*: SASL_SSL with SCRAM-SHA-512 (`cdc-user` credentials from Kubernetes secret)
* *YOLO Serving*: Cluster-internal access only (no external route)
* *LLM endpoint*: KServe with ServiceAccount token injection
* *Inter-namespace traffic*: All Kafka connections use TLS with CA certificate from `cdc-cluster-cluster-ca-cert`

== Configuration

The PPE module is toggled via Helm values:

[source,yaml]
----
ppe:
  enabled: true
  endpoint: "http://yolo-ppe-serving.neuroface.svc:8000"
  classes: "hardhat,safety-vest,goggles"
  confidence: "0.5"
  kafka:
    enabled: true
    bootstrap: "cdc-cluster-kafka-bootstrap.kafka-cdc.svc:9093"
    user: "cdc-user"
    topic: "cv.ppe.detections"
    secretName: "cdc-user"
    caCertSecret: "cdc-cluster-cluster-ca-cert"
----

All parameters are validated by `values.schema.json` and exposed as environment variables to the backend via ConfigMap.
