== Developer Hub

Open link:https://backstage-developer-hub-developer-hub.{cluster_domain}[Red Hat Developer Hub^] for software templates, documentation, and golden paths into this demo stack.

Use *Create…* to scaffold repositories and open them in OpenShift Dev Spaces (or compatible editors) with opinionated tooling.

== How it Works

Developer Hub (based on Backstage) acts as an *Internal Developer Platform (IDP)* that standardizes how developers interact with the CDC pipeline stack:

. A developer opens the *Create* page and selects a software template.
. The template scaffolds a new Git repository (in Gitea or GitHub) with opinionated project structure, devfile, and CI/CD configuration.
. The new component is registered in the Backstage *Catalog* automatically, providing visibility across the organization.
. The developer clicks *Open in Dev Spaces* to launch a cloud-based IDE workspace pre-configured with the right tools, extensions, and credentials.
. Inside Dev Spaces, the developer can edit Camel routes visually with Kaoto, modify SonataFlow workflows, or write Kafka consumers — all connected to the live cluster.

Templates exposed in this environment commonly include:

Exact titles depend on the catalog configured in Developer Hub; search the template list if labels differ slightly.

== Template anatomy

Each software template includes:

[source,yaml]
----
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: camel-kaoto-template
  title: Camel Kaoto Route
  description: Scaffold a Camel route project with Kaoto visual editor
  tags: [camel, kaoto, kafka, cdc]
spec:
  owner: platform-team
  type: service
  parameters:
    - title: Project details
      properties:
        name:
          title: Project name
          type: string
        namespace:
          title: OpenShift namespace
          type: string
          default: kafka-cdc
  steps:
    - id: fetch
      name: Fetch skeleton
      action: fetch:template
      input:
        url: ./skeleton
    - id: publish
      name: Publish to Gitea
      action: publish:gitea
    - id: register
      name: Register in catalog
      action: catalog:register
----

The `skeleton/` directory contains the project structure: `pom.xml`, `devfile.yaml`, Camel route stubs, and Kubernetes manifests.

== Devfile: workspace definition

A minimal `devfile` pins the stack image, resource limits, and starter commands:

[source,yaml]
----
schemaVersion: 2.2.2
metadata:
  name: cdc-demo-workspace
  displayName: CDC Demo Workspace
components:
  - name: tools
    container:
      image: registry.redhat.io/devspaces/udi-rhel8:latest
      memoryLimit: 2Gi
      cpuLimit: "1"
      mountSources: true
      env:
        - name: KAFKA_BOOTSTRAP
          value: cdc-cluster-kafka-bootstrap.kafka-cdc.svc:9092
        - name: APICURIO_URL
          value: http://apicurio-registry-service.kafka-cdc.svc:8080/apis/registry/v2
commands:
  - id: build
    exec:
      component: tools
      commandLine: "mvn package -DskipTests"
      workingDir: ${PROJECT_SOURCE}
  - id: run
    exec:
      component: tools
      commandLine: "mvn quarkus:dev"
      workingDir: ${PROJECT_SOURCE}
  - id: test
    exec:
      component: tools
      commandLine: "mvn verify"
      workingDir: ${PROJECT_SOURCE}
----

=== DevWorkspace resource

When a developer clicks *Open in Dev Spaces*, the operator creates a `DevWorkspace` CR in the user's namespace:

[source,yaml,subs=attributes+]
----
apiVersion: workspace.devfile.io/v1alpha2
kind: DevWorkspace
metadata:
  name: cdc-demo-workspace
  namespace: user1-devspaces
spec:
  started: true
  routingClass: che
  template:
    projects:
      - name: camel-cdc-route
        git:
          remotes:
            origin: https://gitea.{cluster_domain}/user1/camel-cdc-route.git
    components:
      - name: tools
        container:
          image: registry.redhat.io/devspaces/udi-rhel8:latest
          memoryLimit: 2Gi
----

The Dev Spaces operator manages the workspace lifecycle: pod scheduling, storage provisioning, IDE startup, and Git credential injection.

== Pre-configured extensions

Workspaces launched from these templates typically bundle editor extensions such as:

* YAML and Kubernetes manifest support (validation, snippets, schema-aware completion)
* Quarkus and Java tooling where applicable (debugging, live reload)
* Kaoto for Camel route visualization (drag-and-drop editor for integration patterns)
* SonataFlow or workflow editors when the template targets BPM
* OpenShift Toolkit for deploying directly from the IDE

The exact set is defined in the Dev Spaces `DevWorkspace` / devfile for each template; verify in the workspace *Extensions* view after launch.

== Workshop: extend the SonataFlow workflow

The *SonataFlow BPM* template scaffolds a minimal workflow so you can extend it as a hands-on exercise.

=== 1. Scaffold the component

In Developer Hub click *Create* and select the **SonataFlow BPM** template. Fill in the project name (e.g. `my-sonataflow-workflow`) and namespace. The template publishes a Gitea repo and registers the component in the catalog.

=== 2. Open in DevSpaces

From the component page click *Open in Dev Spaces*. The IDE opens with the project cloned and Quarkus tooling ready.

=== 3. View the simple workflow

Open `workflows/customer-onboarding.sw.yaml`. The built-in SonataFlow editor shows a minimal diagram with just a few states.

=== 4. Replace with the full workflow

Replace the content of `customer-onboarding.sw.yaml` with the following complete definition. The visual editor updates automatically.

[source,yaml]
----
id: customer-onboarding
version: "1.0.0"
specVersion: "0.8"
name: Customer Onboarding
description: Customer onboarding workflow triggered by CDC events
start: ReceiveCDCEvent
events:
  - name: cdcCustomerEvent
    source: cdc.public.customers
    type: cdc.customer.created
states:
  - name: ReceiveCDCEvent
    type: event
    onEvents:
      - eventRefs:
          - cdcCustomerEvent
        eventDataFilter:
          data: "${ .customer }"
    transition: ValidateCustomerData

  - name: ValidateCustomerData
    type: operation
    actions:
      - name: validateFields
        functionRef:
          refName: validateCustomer
          arguments:
            customer: "${ .customer }"
    transition: CheckValidation

  - name: CheckValidation
    type: switch
    dataConditions:
      - condition: "${ .validationResult.valid == true }"
        transition: CheckApprovalRequired
      - condition: "${ .validationResult.valid == false }"
        transition: HandleInvalidData
    defaultCondition:
      transition: HandleInvalidData

  - name: HandleInvalidData
    type: operation
    actions:
      - name: notifyInvalid
        functionRef:
          refName: sendNotification
          arguments:
            type: validation_failed
            customer: "${ .customer }"
            errors: "${ .validationResult.errors }"
    end: true

  - name: CheckApprovalRequired
    type: switch
    dataConditions:
      - condition: "${ .customer.creditLimit > 10000 }"
        transition: RequestManagerApproval
    defaultCondition:
      transition: ProvisionAccount

  - name: RequestManagerApproval
    type: operation
    actions:
      - name: autoApprove
        functionRef:
          refName: evaluateApproval
          arguments:
            customer: "${ .customer }"
    transition: EvaluateApproval

  - name: EvaluateApproval
    type: switch
    dataConditions:
      - condition: "${ .approvalResult.approved == true }"
        transition: ProvisionAccount
      - condition: "${ .approvalResult.approved == false }"
        transition: NotifyRejection
    defaultCondition:
      transition: NotifyRejection

  - name: NotifyRejection
    type: operation
    actions:
      - name: sendRejection
        functionRef:
          refName: sendNotification
          arguments:
            type: rejected
            customer: "${ .customer }"
            reason: "${ .approvalResult.reason }"
    end: true

  - name: ProvisionAccount
    type: operation
    actions:
      - name: createAccount
        functionRef:
          refName: provisionBankAccount
          arguments:
            customer: "${ .customer }"
      - name: setupOnlineBanking
        functionRef:
          refName: provisionOnlineBanking
          arguments:
            customer: "${ .customer }"
    transition: EmitOnboardingMetric

  - name: EmitOnboardingMetric
    type: operation
    actions:
      - name: emitMetric
        functionRef:
          refName: sendNotification
          arguments:
            type: onboarding_complete
            customer: "${ .customer }"
    end: true

functions:
  - name: validateCustomer
    operation: specs/customer-service.yaml#validateCustomer
    type: rest
  - name: evaluateApproval
    operation: specs/customer-service.yaml#evaluateApproval
    type: rest
  - name: provisionBankAccount
    operation: specs/customer-service.yaml#provisionBankAccount
    type: rest
  - name: provisionOnlineBanking
    operation: specs/customer-service.yaml#provisionOnlineBanking
    type: rest
  - name: sendNotification
    operation: specs/notification-service.yaml#sendNotification
    type: rest
----

=== 5. Run locally

In the Dev Spaces terminal run:

[source,bash]
----
mvn quarkus:dev
----

This starts the Quarkus dev mode. The SonataFlow engine loads the workflow and exposes it at `http://localhost:8080/customer-onboarding`.

=== 6. Commit and push

Once satisfied with the changes, commit and push back to Gitea. ArgoCD or the CI pipeline picks up the changes for deployment to the cluster.

== Resource governance

Dev Spaces workspaces run inside the OpenShift cluster and are subject to resource quotas:

[cols="1,1,2",options="header"]
|===
|Resource |Default limit |Purpose

|*Memory*
|2Gi per workspace
|Prevents individual workspaces from consuming excessive memory

|*CPU*
|1 core per workspace
|Ensures fair scheduling across multiple concurrent developers

|*Storage*
|10Gi per workspace
|PVC for project sources, caches, and built artifacts

|*Concurrent workspaces*
|1 per user
|Controlled by Dev Spaces operator configuration
|===

Administrators can adjust these limits via the `CheCluster` CR or per-namespace `ResourceQuota` objects.
