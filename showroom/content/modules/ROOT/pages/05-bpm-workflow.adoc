=== Why BPM in a CDC Pipeline?

[cols="1,1",options="header"]
|===
|Without BPM |With BPM (SonataFlow)

|CDC events are processed linearly
|CDC events trigger business processes with conditional logic

|No approvals or SLAs
|Approvals with timeout (24h SLA for premium customers)

|Simple notifications (fire-and-forget)
|Parallel orchestration: provision + notify simultaneously

|No customer classification
|Automatic classification: basic / standard / premium

|No process metrics
|Onboarding metrics emitted to Kafka
|===

  name: logic-operator-rhel8

  channel: alpha
  installPlanApproval: Automatic
  name: logic-operator-rhel8

* `SonataFlow` вҖ” Defines a serverless workflow

== Workflow states

[cols="2,1,3",options="header"]
|===
|State |Type |Description

|`ReceiveCDCEvent`
|Event
|Listens for CDC events from `cdc.public.customers` via CloudEvents

|`ValidateCustomerData`
|Switch
|Validates that the event contains `email` and `first_name`

|`ClassifyCustomer`
|Operation
|Classifies customer tier: basic (<10k), standard (10k-50k), premium (>50k)

|`CheckApprovalRequired`
|Switch
|Evaluates whether the tier requires manual approval

|`RequestManagerApproval`
|Callback
|Sends approval request to `bpm.approval.requests` via Kafka Bridge. Waits for response event with 24-hour timeout

|`EvaluateApproval`
|Switch
|Evaluates the approval response

|`ProvisionAccount`
|Parallel
|Runs in parallel: create account + send welcome email via Mailpit

|`EmitOnboardingMetric`
|Operation
|Publishes onboarding metric to `bpm.onboarding.metrics`
|===

[source,yaml]
----
      - name: RequestManagerApproval
        type: callback
        action:
          name: requestApproval
          functionRef:
            refName: sendApprovalRequest
          actionDataFilter:
            fromStateData: |
              {
                "records": [{
                  "value": {
                    "customer_id": .customer.id,
                    "name": "\(.customer.first_name) \(.customer.last_name)",
                    "email": .customer.email,
                    "tier": .tier,
                    "status": "pending_approval"

                }]

== How it Works

=== SonataFlow execution engine

SonataFlow executes workflows defined in the https://serverlessworkflow.io/[Serverless Workflow Specification] as Quarkus microservices:

. The operator receives the `SonataFlow` CR and generates a *Quarkus application* with the embedded workflow runtime.
. Upon receiving a CDC event (CloudEvent via Kafka or HTTP), the engine creates a *workflow instance* with a unique ID.
. Each instance maintains its own *state data* вҖ” a JSON document that transforms at each transition.
. The engine executes transitions defined in the workflow in an *event-driven* manner: each state is evaluated and executed in order, using the previous result as input.

=== Callback pattern: asynchronous approvals

The `RequestManagerApproval` state uses the specification's *callback pattern*:

. The workflow executes the *action* (send approval request to `bpm.approval.requests` via Kafka Bridge).
. The engine *suspends* the workflow instance вҖ” it consumes no CPU or memory while waiting.
. The instance is persisted in the Data Index with status `WAITING`.
. When the `approvalCompleted` event arrives (a message in the topic with the correct `workflowInstanceId`), the engine *resumes* the instance from the exact point where it was suspended.
. If the event does not arrive within `PT24H` (24 hours), the timeout fires and the workflow transitions to the next state with a timeout flag.

=== Parallel execution: branches

The `ProvisionAccount` state executes *branches* in parallel:

. The engine launches both branches (`CreateAccount` and `SendWelcome`) simultaneously as coroutines.
. `completionType: allOf` indicates the workflow waits for *both* branches to complete before transitioning.
. If a branch fails, the workflow can retry or handle the error according to the defined policy.
. Results from both branches are merged into the instance's state data.
