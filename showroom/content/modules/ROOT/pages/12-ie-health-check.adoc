= Step 10: Industrial Edge — Health Check

Before demonstrating any Industrial Edge scenarios, verify that all components are running and healthy.

== 1. Check ArgoCD applications

Open the OpenShift GitOps (ArgoCD) console and confirm that all `industrial-edge-*` applications are *Synced* and *Healthy*:

[source,bash,subs=attributes+]
----
oc get applications -n openshift-gitops | grep industrial-edge
----

Expected output (all should show `Synced` and `Healthy` or `Progressing`):

[source,text]
----
field-content-industrial-edge-data-lake              Synced   Healthy
field-content-industrial-edge-data-science-cluster   Synced   Healthy
field-content-industrial-edge-data-science-project   Synced   Healthy
field-content-industrial-edge-minio                  Synced   Healthy
field-content-industrial-edge-pipelines              Synced   Healthy
field-content-industrial-edge-stormshift             Synced   Healthy
field-content-industrial-edge-tst                    Synced   Healthy
----

== 2. Verify operators

Confirm that the required operators are installed and running:

[source,bash]
----
oc get csv -n industrial-edge-tst-all | grep -E "amq-broker|amq-streams|camel-k"
oc get csv -n redhat-ods-operator | grep rhods
----

You should see `Succeeded` for each operator.

== 3. Check the seed pipeline

The initial seed pipeline in namespace `industrial-edge-ci` builds all required container images into the internal registry. Verify it ran successfully:

[source,bash]
----
oc get pipelineruns -n industrial-edge-ci
----

If the seed pipeline has not run or has failed, you can trigger it manually:

[source,bash]
----
oc create -f examples/helm/components/industrial-edge-pipelines/extra/seed-run.yaml -n industrial-edge-ci
----

== 4. Verify the Line Dashboard

The Line Dashboard is the primary UI for the Industrial Edge demo. Access it via its OpenShift Route:

[source,bash,subs=attributes+]
----
oc get route line-dashboard -n industrial-edge-tst-all -o jsonpath='{.spec.host}'
----

Open the URL in your browser. Click on *Realtime Data* in the left navigation and wait a moment. You should see:

* *Vibration data* streaming from the simulated machine sensors
* After 2–3 minutes, an *anomaly alert* generated by the ML inference model

NOTE: Initially, only vibration data may appear. Temperature data will be enabled in the xref:13-ie-config-gitops.adoc[GitOps configuration demo].

== 5. Verify sensor pods

Confirm the machine sensor pods are running and generating data:

[source,bash]
----
oc get pods -n industrial-edge-tst-all -l app=machine-sensor
----

Check sensor logs to verify MQTT publishing:

[source,bash]
----
oc logs -n industrial-edge-tst-all -l app=machine-sensor --tail=10
----

== 6. Verify Kafka topics

Check that Kafka topics are receiving sensor data:

[source,bash]
----
oc get kafkatopics -n industrial-edge-tst-all
----

You should see topics for temperature and vibration data. Verify messages are flowing:

[source,bash]
----
oc exec -n industrial-edge-tst-all industrial-edge-kafka-cluster-kafka-0 -- \
  bin/kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic iot-sensor-sw-vibration \
  --from-beginning --max-messages 3
----

== 7. Verify MinIO

MinIO provides S3-compatible storage for the data lake. Check its Route and access the console:

[source,bash,subs=attributes+]
----
oc get route minio-console -n industrial-edge-ml-workspace -o jsonpath='{.spec.host}'
----

Login with:

* *Username:* `minioadmin`
* *Password:* `minioadmin`

Verify that the following buckets exist:

* `pipeline-bucket` — Used by ML training pipelines
* `user-bucket` — Used for model artifacts
* `anomaly-models` — Stores trained anomaly detection models

== 8. Verify OpenShift AI

Check that the Data Science Cluster is ready:

[source,bash]
----
oc get datasciencecluster -A
----

Verify the ML workspace project exists:

[source,bash]
----
oc get project industrial-edge-ml-workspace
oc get pods -n industrial-edge-ml-workspace
----
